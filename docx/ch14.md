## 필수 검증 순서 (순서대로)

심사자는 아래 순서대로 항목을 확인해 주세요. 각 항목은 문서의 해당 체크리스트 항목과 연결됩니다.

1. GitHub 저장소 URL 제출 및 접근 가능한지 확인 (제출물)
2. Vercel 배포 URL 제출 및 사이트 접속 확인 (제출물, F1)
3. `middleware.ts`가 프로젝트 루트에 존재하는지 확인 (A7)
4. `AuthProvider` 또는 인증 컨텍스트가 존재하는지 확인 (A7)
5. `lib/supabase.ts` 또는 `lib/supabase/client.ts`가 존재하는지 확인 (E3)
6. `supabase/migrations/` 아래 RLS 마이그레이션 파일이 있는지 확인 (C4)
7. `tests/` 폴더에 Playwright 테스트 파일이 있는지 확인 (F2)
8. Playwright 테스트가 로컬 또는 CI에서 실행되어 실패가 없는지 확인 (권장 — F3와 연계하여 검증)

## G. 보너스 (최대 5점)

아래 항목별로 구현/질이 좋을수록 점수를 부여합니다. 항목당 기본 1점, 추가 구현·완성도로 최대 합계 **5점**까지 인정합니다.

- **G1 댓글 (확인‧파일)**: 댓글 UI 및 API 존재 (`app/components/CommentSection.tsx`, `lib/comments.ts`). 배포 사이트에서 게시글 상세에 댓글 작성/삭제 동작 확인. (1점)
- **G2 좋아요 / 추천 (확인‧권장 구현)**: 좋아요 버튼과 카운터 또는 likes 테이블이 있어야 함. 권장: `likes` 테이블 또는 `posts.likes_count` 필드 + API 엔드포인트/클라이언트 처리. 배포에서 버튼 클릭 시 즉시 증가하고 재요청 시 유지되는지 확인. (1점)
- **G3 프로필 페이지 (확인‧파일)**: `/profile` 또는 `/mypage`에서 사용자 정보(이메일/username/avatar)와 본인 글 목록이 보여야 함 (`app/mypage/page.tsx`). (1점)
- **G4 이미지 업로드 (확인‧권장 구현)**: 글 작성 또는 프로필에서 이미지 업로드가 가능해야 함. 권장: Supabase Storage 사용(프리사인 업로드 또는 `supabase.storage.from(bucket).upload()`), 마이그레이션에 `avatar_url` 컬럼 확인. 업로드 후 이미지가 게시글/프로필에 노출되는지 확인. (1점)
- **G5 검색 기능 (확인‧파일)**: 제목/내용/작성자 기준 검색이 동작해야 함 (`app/components/SearchBar.tsx`). 배포에서 키워드 입력 시 결과 필터링 확인. (1점)
- **G6 다크 모드 (확인‧권장 구현)**: 다크 스타일을 제공하고(예: `.dark` 클래스 기반) 토글 UI 또는 ThemeContext로 전환 가능하면 가산. 권장: `ThemeProvider` + `ThemeToggle` 컴포넌트. (가산 점수 — 구현 수준에 따라 0~1점)
- **G7 반응형 디자인 (확인)**: 모바일에서 레이아웃 정상 동작(예: `grid-cols-1 md:grid-cols-2`). (판정 보조)
- **G8 기타 (확인)**: 수업 범위를 벗어난 추가 기능(실시간, 알림 등). (가산)

체크리스트(권장 심사 흐름):

1. 배포 사이트에서 G1~G5를 우선 확인한다(댓글, 검색, 프로필, 이미지, 좋아요).  
2. G2/G4/G6는 구현이 복잡하므로 코드 확인으로 보완한다(예: `lib/`에 관련 함수, `supabase/migrations`에 컬럼, Storage 호출).  
3. 구현이 부족하면 권장 보강안(간단한 likes 테이블, Supabase Storage 업로드, ThemeContext 토글)을 제안하고 감점 기준을 명시한다.
